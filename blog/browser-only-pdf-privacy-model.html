Browser-Only PDF: Our Privacy & Threat Model

How your PDFs are processed 100% locally, what data never leaves your device, and how to verify it with your own browser.

Updated: 2025-11-12 · 6 min read

What “browser-only” means

In DocPDFHub, all PDF operations run inside your browser. We use client-side libraries (e.g., PDF.js and PDF-Lib) and Web APIs like Canvas, WebAssembly, and FileReader. Your file bytes are read from your disk into browser memory, processed locally, and exported back to disk—never uploaded to our servers.

Data flow at a glance

  1. File selection: You pick files via the file picker or drag-and-drop. The browser gives our page a handle to read bytes locally.
  2. Local processing: Pages render on an off-screen canvas; compression and merging happen using JavaScript/WebAssembly.
  3. Output: We create a Blob/ZIP/PDF and trigger a local download. Temporary data lives in RAM, not on our servers.

Threat model

VectorWhat we doWhat you can do
Server upload We never send file bytes to our server. Verify with DevTools → Network: filter by document|xhr|fetch while converting.
Third-party beacons No analytics on tool steps that process bytes. Only essential scripts (fonts/CSS) load. Use content blockers if desired; compare outcomes—should be identical.
Browser storage We do not persist file content in localStorage/IndexedDB. Refresh page to clear in-memory buffers. Avoid running many heavy tabs simultaneously.
Memory scraping on shared device Buffers stay in tab memory; we release object URLs after download. Close the tab after use; use a private window on shared computers.
Malicious PDFs Processing is sandboxed by the browser; we don’t execute embedded code. Only open files you trust. Keep your browser up to date.

What never leaves your device

  • PDF file bytes and page pixel data
  • Rendered canvases and intermediate images
  • Generated outputs (compressed PDFs, JPGs, ZIP files)

What may go over the network

  • Static assets: HTML/CSS/JS from our site or CDNs (loaded before you add files)
  • Optional fonts and icons
  • Ads/analytics (if present) — but never file bytes or page pixels

How to verify locally

  1. Open DevTools → Network.
  2. Click Select PDF, process the file, and watch requests.
  3. You should only see local blob: URLs for outputs and static asset requests made before conversion.
  4. No XHR/Fetch upload of your file should appear.

Limits and tips

  • Large files: Browsers are memory-bounded. If you hit limits, try splitting the PDF first.
  • Mobile: Use “Balanced/Fast” quality to reduce RAM usage.
  • Offline: After first load, the tools work offline if cached by your browser.

Compliance considerations

Because documents never leave your device, there’s no transfer of personal data to our servers during processing. For regulated environments, confirm with your security team whether local-only tooling meets your requirements.

FAQ

Q: Do you store my files?
A: No. Files stay in your browser memory only.

Q: What about the output I download?
A: Downloads are written directly to your disk by the browser.

Q: Can I run the tools offline?
A: Yes, after the first load if your browser cached the assets.
